# Docker Workflow Instructions

## Naming convention (env-driven, avoids collisions between stacks/envs on the same host)

- `APP_NAME` = short app slug (e.g. `myapp`). `ENV_NAME` = `local` | `production` | ...
- `STACK_NAME=${APP_NAME}-${ENV_NAME}` — used as a Traefik constraint label so multiple stacks/envs can share one Docker host without routing collisions.
- Every `container_name` / `hostname` follows `${APP_NAME}-<service>-${ENV_NAME}` (e.g. `myapp-postgres-local`), except the main app container which is just `${APP_NAME}`.
- Networks are named explicitly (not left to compose defaults): `${APP_NAME}-internal-${ENV_NAME}` and `${APP_NAME}-external-${ENV_NAME}`.
- Image tags: `<org>/<service>[-<subservice>]:${VERSION}` plus a floating `:latest` (`<org>` = your container registry namespace/org). `VERSION` for the main app is read from `package.json` at build time (`node -p "require('./package.json').version"`), not from the env file. Compose files default `VERSION` to `latest` via `${VERSION:-latest}`.

## File layout

```
docker/
  docker-compose.yml              # dev: builds every service from source, bind-mounts code
  docker-compose.prod.yml         # prod: uses pre-built images, no source bind-mounts
  docker-compose-admin.prod.yml   # admin is deployed independently in prod → its own compose file
  conf/
    .env.dev.example              # committed template, blank secrets
    .env.dev                      # gitignored, generated by scripts/setup-env.sh
    .env.prod                     # committed, blank secrets filled in at deploy time
```
Each buildable sub-app (root API, `admin/`, `website/`, `mocks/*`, `db/liquibase/`, `db/pgsql/`) owns its own `dockerfile` (dev) and `dockerfile.prod` (prod) next to its source, referenced by `build.context`/`build.dockerfile` in the compose files — not a single shared dockerfile.

## Compose YAML anchors (DRY across services)

Define shared fragments once at the top of the compose file and reuse with `<<: *name`:
- `x-service-healthcheck: &healthcheck` — reused via `healthcheck: *healthcheck` on any service needing the same check.
- `x-secret-args: &secretArgs` — the list of build secrets (`apt_repository`, `apk_repository`, `npmrc`) merged into a service's `build:` block with `<<: *secretArgs`.
- `x-secret-mount: &secretMount` — runtime secret mounts (e.g. `.npmrc` at `/home/user/.npmrc`) merged via `secrets: *secretMount`.
- `x-default-args: &defaultArgs` / `x-default-envs: &defaultEnvs` — common build args (`UID`, `GID`, `TZ`) / env vars merged with `<<: *defaultArgs`.

## Dockerfile pattern — dev vs prod

**Dev dockerfile** (single stage, optimized for live-reload):
- `FROM node:${NODE_VERSION}`, install full deps (`npm i --ignore-scripts --no-fund`), `COPY` only `package*.json` (source is bind-mounted by compose, not copied).
- `CMD [ "node", "--run", "dev" ]`.

**Prod dockerfile** (multi-stage):
- Stage `deps`: `npm ci --only=production --ignore-scripts --no-fund && npm cache clean --force`.
- Final stage: copy `node_modules` from `deps`, copy `package.json` + `src` only, then **remove `npm`/`npx`/`corepack` binaries** from the final image (`rm -rf /usr/local/lib/node_modules/npm /usr/local/lib/node_modules/corepack /usr/local/bin/npm /usr/local/bin/npx /usr/local/bin/corepack`) since the app is started via `node --run start`, not `npm`.
- Frontend apps (Angular admin, VitePress website) use a build stage (`node:...` → `npm run build:prod`) + a final `nginx:${NGINX_VERSION}` stage that removes the default nginx conf and copies a custom `nginx/default.conf`, then `COPY --from=build /usr/src/app/dist/ /usr/share/nginx/html`.

**Non-root user pattern** (every dockerfile, dev and prod):
```
ARG UID
ARG GID
RUN deluser --remove-home node && addgroup -S usergroup -g ${GID} && adduser -G usergroup -S user -u ${UID}
USER user
```
`UID`/`GID` are passed as build args matching the host user (`${UID}`/`${GID}` env vars) so bind-mounted files created inside the container are owned by the host user, not root.

## Private registry / BuildKit secrets pattern

Used whenever a service needs a private Alpine mirror and/or a private npm registry during build:
```
RUN rm /etc/apk/repositories
RUN --mount=type=secret,id=apk_repository,target=/etc/apk/repositories,required=true apk update && apk add tzdata
...
ARG NPMRC_PATH
RUN --mount=type=secret,id=npmrc,target=${NPMRC_PATH},required=true,uid=${UID}
```
- Compose declares top-level `secrets:` sourced from env vars (`environment: APK_REPOSITORY` / `NPMRC`), not files — so secret content lives only in the env file, never committed.
- `scripts/build-prod.sh` passes them to plain `docker build` via `--secret id=apk_repository,env=APK_REPOSITORY --secret id=npmrc,env=NPMRC`.
- `NPMRC_PATH` build arg = `${HOME_PATH}.npmrc` so the mount target matches the non-root user's home.

## Migration (Liquibase) container pattern

- Custom `dockerfile` extends `liquibase/liquibase:${LIQUIBASE_VERSION}`, adds `postgresql-client` + `curl` as root, then drops back to the `liquibase` user, and replaces the entrypoint with a custom `entrypoint.sh` (copied `chmod=755`, owned by root so the non-root runtime user can't modify it).
- Dev vs prod differ only in changelog delivery: dev bind-mounts the changelog folder (e.g. `./changelog`) and `./data` as volumes; prod `COPY`s the changelog folder into the image (baked changelog) and exposes `/liquibase/data` as an optional `VOLUME` for runtime overrides.
- `entrypoint.sh` dispatches on env vars, not CLI args: if `$1` is a recognized OS command it's exec'd directly (passthrough for debugging); otherwise behavior is `UPDATE=1` → create db + `update` + apply optional `/liquibase/data/changelog.xml` + snapshot + `create_user` (idempotent `psql` checks via `SELECT 1 FROM pg_roles/pg_database` before `CREATE`), `ROLLBACK>1` → `rollback-count` + snapshot, else → `diff-changelog` + `changelog-sync`.
- Snapshot/log file generation are opt-in via `LIQUIBASE_ENABLE_SNAPSHOT` / `LIQUIBASE_ENABLE_LOG_FILE` (default off) to avoid polluting the image/host with local artifacts.
- App DB user creation grants are scoped per-schema (`public` + `log`) rather than blanket superuser rights.

## Postgres container pattern

- Custom `dockerfile` just extends the official postgres image and `COPY`s a `create_db.sh` into `/docker-entrypoint-initdb.d/` to create multiple databases from a comma-separated `POSTGRES_DBS` env var (official image only creates one DB by default).

## Traefik / routing pattern

- One `traefik` service per stack, constrained to only its own stack via `--providers.docker.constraints=Label(\`stack.name\`,\`${APP_NAME}-${ENV_NAME}\`)` — required so multiple app stacks can run on the same Docker daemon without cross-routing.
- Every routed service gets labels: `traefik.enable=true`, `stack.name=${STACK_NAME}`, a `PathPrefix` rule per mount path (e.g. `/admin`, `/swagger`, `/docs`, `/adminer`), a single `web` entrypoint, and (for services expecting requests without the prefix) a `stripprefix` middleware.
- Dev-only: `--api.insecure=true` + `--api.dashboard=true` expose the Traefik dashboard at `/dashboard`; disabled in prod compose.

## Service startup ordering

Use `depends_on` with explicit conditions rather than plain lists: `postgres: condition: service_healthy` gates the migration container; `postgres: service_healthy` + `<app>_migration: condition: service_completed_successfully` gates the app container.

## Dev bind-mounts

- Mount individual files/dirs needed for live-reload (`src`, `tests`, `jest.config.js`, `babel.config.json`, `package.json`) rather than the whole project root, to avoid clobbering `node_modules`.
- `node_modules` for services with a host-side `npm install` step (main app, admin) use `type: bind` with `bind.create_host_path: true` pointing at the host's `node_modules` dir — `scripts/start-dev.sh` does `mkdir -p node_modules admin/node_modules` before `up` so Docker doesn't create these root-owned.
- Services without a host-side install (swagger, website, mocks) use plain named volumes for `node_modules` instead.

## `scripts/*.sh` conventions

- `setup-env.sh`: copies `.env.dev.example` → `.env.dev`, generates secrets with `openssl rand -base64 24` / `openssl rand -hex 4`, applies them with a cross-platform `sedi()` helper (`sed -i ''` on `darwin`, `sed -i` elsewhere), then re-sources the generated file (`source ... ; set +e/-e`) to conditionally build an `NPMRC` value only when a registry URL is configured.
- `start-dev.sh` / `stop-dev.sh`: always pass both `-f docker/docker-compose.yml` and `--env-file docker/conf/.env.dev` explicitly (never rely on compose auto-discovery). `stop-dev.sh` accepts `--rmi`/`-i` to also remove images, and always explicitly removes the named `postgres_data` volume afterward (compose `down` doesn't remove named volumes by default).
- `build-prod.sh`: reads config from `.env.prod` via `grep -E "^${VAR}=" file | cut -d'=' -f2-` helper instead of `source`-ing the file, specifically to avoid clashing with shell-readonly vars like `UID`. Accepts optional positional args (one per buildable service, e.g. `api migration admin website`) to build a subset; builds/tags every image with both `${VERSION}` (from `package.json`) and `latest`.
- `start-prod.sh`: reads `VERSION` from `.env.prod` the same grep way and exports it before `docker compose -f docker/docker-compose.prod.yml --env-file docker/conf/.env.prod up -d`.
- `reset-*.sh` (targeted teardown for fast iteration on one service): source the env file while excluding the `UID=` line (`grep -v '^UID='`) to avoid overwriting the shell's read-only `$UID`; remove container + matching images (`docker images --format ... | grep -E "(<org>/)?${APP_NAME}.*<service>"`) + the service-specific volume, then restart via the corresponding `start-*.sh`.

## `.dockerignore`

Excludes tests (`tests/`, `test/`, `*.test.js`, `*.spec.js`), `docs/`, `.github/`, VCS/log/coverage artifacts — keep build context minimal per sub-app.
